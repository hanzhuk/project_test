# Go-Scaffold 脚手架系统全流程使用说明文档

## 一、 系统能力概览

### 1.1 系统能做到什么？
`Go-Scaffold` 是一款基于大模型（LLM Tool Calling）与 Go 模板驱动的现代化后端项目生成与代码演进脚手架。它能够：
1. **一键生成现代化后端基础架构**：自动搭建包含 Web 框架（Echo / Gin / Fiber）、ORM（Ent / GORM）、数据库（PostgreSQL / MySQL / SQLite）、Slog 结构化日志、Docker 开发环境的标准 Go 后端应用。
2. **自然语言/TUI 驱动业务模块代码生成**：通过自然语言描述（如 `Book 图书模块，包含 title:string, author:string, price:float`）或 TUI 表单，利用本地大模型自动生成标准的 Ent Schema、Data Handler、Service 层代码及路由注册。

### 1.2 做到什么程度？
- **零配置开箱即用**：生成的项目自带完整的 `go.mod`、`Dockerfile`、`docker-compose.yml` 及数据库初始化 Migration 文件。
- **高质量代码工程规范**：严格遵循分层架构设计，所有生成代码均包含完整的错误处理、结构化上下文日志（Slog）及标准健康检查接口（`/ping`、`/health`）。
- **抗幻觉与强类型保障**：内置 Few-Shot 标准范例提示词约束，避免大模型虚构不存在的 ORM API 方法，生成代码可通过 Go 静态编译器校验。

### 1.3 最终产出结果
- **脚手架产物**：编译好的二进制可执行文件 `go-scaffold.exe`。
- **生成的后端项目**：具备完整工程结构与业务 CRUD 接口的 Go 服务，可直接运行、连接数据库并响应 HTTP API 请求。

## 二、 全流程项目启动指南

### 第一步：编译脚手架可执行文件 (`go-scaffold.exe`)

#### 1. 操作指令
在脚手架源码根目录（即 `g:\project_test\project_test`）下打开终端，执行：
```bash
go build -o go-scaffold.exe ./cmd/go-scaffold
```

#### 2. 执行后会发生什么？
- Go 编译器将 `cmd/go-scaffold/main.go` 及其依赖项编译为可执行文件。
- 在当前目录下生成 `go-scaffold.exe`。

#### 3. 结果与验证
在终端输入：
```bash
.\go-scaffold.exe --help
```
若输出了脚手架的帮助信息（包含 `init`、`generate`、`version` 等命令提示），则说明脚手架编译成功。

---

### 第二步：准备本地 Ollama AI 大模型环境

#### 1. 操作指令
启动 Ollama 服务并拉取推荐的代码专用模型：
```bash
# 1. 确保 Ollama 后台服务正常运行
ollama serve

# 2. 拉取代码专精模型（强烈推荐）
ollama pull qwen2.5-coder:7b
```

#### 2. 执行后会发生什么？
- Ollama 将从本地或网络加载 `qwen2.5-coder:7b` 模型到内存/显存中。
- 脚手架后续进行代码生成时，将优先通过 Ollama API 调用此模型。

---

### 第三步：初始化全新的全栈项目 (`init`)

#### 1. 操作指令
在存储项目的目录下运行 `init` 命令，指定带 React 前端（以创建名为 `my-app` 的全栈应用为例）：

**非交互模式（命令行一键生成全栈应用）：**
```bash
.\go-scaffold.exe init my-app --backend echo --orm ent --db postgres --frontend react
```

**交互模式（TUI 可视化终端）：**
```bash
.\go-scaffold.exe init my-app
```

#### 2. 执行后会发生什么？
- 脚手架根据内置模版渲染出标准的 Go + React 全栈目录结构。
- 在 `my-app` 根目录下自动创建项目元数据配置文件 `.go-scaffold.json`。
- 生成包含 Echo 路由、Ent ORM、Slog 日志、`/ping` 与 `/health` 健康检查接口的后端服务，以及位于 `web/` 目录下的 React + Vite 前端工程。

#### 3. 产出的项目目录结构：
```text
my-app/
├── .go-scaffold.json        # 项目元数据（记录后端与前端技术栈）
├── docker-compose.yml       # 数据库 Docker 容器配置
├── go.mod                   # Go 依赖清单
├── cmd/
│   └── server/
│       └── main.go          # 后端服务启动入口
├── ent/
│   └── schema/              # Ent ORM 数据模型 Schema
├── internal/
│   └── handler/             # HTTP 接口处理函数与路由注册
└── web/                     # React + Vite 前端工程目录
    ├── package.json         # 前端 npm 依赖配置文件
    ├── vite.config.ts       # Vite 构建配置
    └── src/                 # React 页面组件源码
```

---

### 第四步：使用 AI 快速生成业务模块代码 (`generate`)

#### 1. 操作指令
进入刚生成的项目目录 `my-app`，使用 AI 命令生成图书业务模块：

```bash
cd my-app

# 使用根目录编译好的脚手架进行业务代码生成
..\go-scaffold.exe generate "Book 图书模块，包含 title:string, author:string, price:float" --no-tui --model qwen2.5-coder:7b
```

#### 2. 执行后会发生什么？
- 脚手架自动读取 `my-app/.go-scaffold.json` 获取当前项目的技术栈（Echo + Ent + PostgreSQL）。
- 脚手架组装标准的 System Prompt，将自然语言需求推送到 Ollama 模型。
- 模型通过 Tool Calling 触发文件写入，在 `ent/schema/book.go` 中自动生成标准的 Ent 数据表 Schema。

---

### 第五步：编译 Ent 客户端与依赖整理

#### 1. 操作指令
在 `my-app` 目录下依次运行：

```bash
# 步骤 1：根据 ent/schema/*.go 自动生成 Ent ORM 客户端代码
go run -mod=mod entgo.io/ent/cmd/ent generate ./ent/schema

# 步骤 2：自动下载并清理 package 依赖项
go mod tidy

# 步骤 3：进行全项目静态编译校验
go build ./...
```

#### 2. 执行后会发生什么？
- **步骤 1**：Ent 编译器读取 `ent/schema/book.go`，在 `ent/` 目录下生成 `book.go`、`client.go`、`mutation.go` 等全套增删改查客户端代码。
- **步骤 2**：Go 模块管理器补齐缺失的底层依赖库。
- **步骤 3**：验证整个应用的所有 Go 文件无语法错误、无包引用缺失。若没有输出报错信息，说明代码 100% 正确！

---

### 第六步：启动全栈服务（后端数据库 + React 前端）

#### 1. 启动 PostgreSQL 数据库与 Go 后端（终端窗口 1）
```bash
# 启动数据库容器
docker compose up -d db

# 启动 Go 后端 API 服务 (监听 8080 端口)
go run cmd/server/main.go
```

#### 2. 启动 React 前端工程 (终端窗口 2)
打开一个新的终端窗口，进入 `my-app/web` 目录：
```bash
# 1. 进入前端工程目录
cd web

# 2. 安装 npm 依赖项
npm install

# 3. 启动 Vite 前端开发服务器 (监听 5173 端口)
npm run dev
```

#### 3. 最终运行效果
- 后端控制台输出 Slog 数据库连接与 HTTP 启动日志：`HTTP 服务启动 addr=:8080`
- 浏览器打开 `http://localhost:5173` 访问 React 前端可视化界面，自动联调后端 API！

## 三、 我该如何验证它？

为了全方位验证系统生成的代码质量与功能完整性，请按照以下三个步骤进行验证：

### 1. 验证静态编译与代码规范 (Static Verification)
在 `my-app` 项目根目录下运行：
```bash
go build ./...
```
**判定标准**：命令执行完成后，终端直接返回新一行提示符，未抛出任何 Go 编译器语法报错。

### 2. 验证健康检查 API (HTTP Verification)
服务启动后（`go run cmd/server/main.go`），打开浏览器或在另一个终端终端运行 `curl` 命令：

```bash
# 验证 API v1 业务 Ping 接口
curl http://localhost:8080/api/v1/ping

# 验证根目录 Health 健康检查接口
curl http://localhost:8080/health
```

**预期返回结果**：
- `/api/v1/ping` 返回：`{"code":0,"message":"success","data":"pong"}`
- `/health` 返回：`{"status":"ok"}`

### 3. 验证数据库 ORM 接口 (DB & CRUD Verification)
检查 `ent/` 目录：
- 确认自动生成了 `ent/book/` 和 `ent/book.go`。
- 确认增删改查客户端方法（`client.Book.Create()`、`client.Book.Query()`）均已准备就绪。

## 四、 全栈模式 vs 纯后端模式说明 (Full-Stack vs Backend-Only)

`Go-Scaffold` 支持 **纯后端 API 模式** 和 **React 全栈模式** 两种项目形态：

### 1. 模式区别

| 模式 | 初始化命令 | 适用场景 | 生成产物 |
| :--- | :--- | :--- | :--- |
| **纯后端 API 模式** | `.\go-scaffold.exe init my-shop --backend echo --orm ent` | 微服务/API 服务 | 仅 Go 后端与 ORM 代码（无需 npm） |
| **React 全栈模式** | `.\go-scaffold.exe init my-app --frontend react` | 全栈 Web 应用 | Go 后端 + `web/` 前端工程（包含 `package.json`） |

---

### 2. 全栈项目的完整启动指令

若在初始化时指定了 `--frontend react`（或在 TUI 中选择了 React 前端），项目根目录下会生成 `web/` 目录。启动全栈应用的步骤如下：

#### 终端窗口 1：启动后端数据库与 API 服务
```bash
# 1. 启动数据库
docker compose up -d db

# 2. 启动 Go 后端服务 (端口 8080)
go run cmd/server/main.go
```

#### 终端窗口 2：启动 React 前端工程 (Vite)
```bash
# 1. 进入前端工程目录
cd web

# 2. 安装前端依赖包
npm install

# 3. 启动前端 Vite 本地开发服务器 (端口 3000 / 5173)
npm run dev
```

启动后在浏览器打开 `http://localhost:5173` 即可访问可视化前端页面，前端将自动通过 API 请求后端 `http://localhost:8080/api/v1`。

## 五、 前端与全栈对接指南 (Frontend Integration)

生成的基础设施已内置前端联调与跨域支持：

### 1. 跨域配置 (CORS Support)
生成的后端在 `cmd/server/main.go` 中默认开启了全域 CORS 中间件：
```go
e.Use(echomw.CORSWithConfig(echomw.CORSConfig{
    AllowOrigins: []string{"*"},
    AllowMethods: []string{http.MethodGet, http.MethodPost, http.MethodPut, http.MethodDelete},
}))
```
前端开发服务（`http://localhost:5173`）可以直接向 Go 后端发送 Fetch/Axios 请求，无需配置复杂的代理即可跨域访问。

### 2. 前端 BaseURL 配置规范
前端 HTTP Client（例如 Axios）建议设置统一的前缀：
```javascript
const api = axios.create({
  baseURL: 'http://localhost:8080/api/v1', // 后端 API v1 统一前缀
  timeout: 5000,
});

// 示例：调用 Ping 接口
api.get('/ping').then(res => console.log(res.data));
```

## 六、 常见报错 FAQ

### Q1: 为什么执行 `curl http://localhost:8080/ping` 会报错 `Not Found` (code: 1002)？
- **原因**：脚手架设计的 RESTful API 采用了版本化路由组。所有的业务接口统一挂载在 `/api/v1` 前缀之下。
- **解决办法**：
  - 请求业务 Ping 接口：`curl http://localhost:8080/api/v1/ping`
  - 请求根路径健康检查：`curl http://localhost:8080/health`

### Q2: 为什么提示 `docker compose up -d db` 拉取镜像超时？
- **原因**：国内访问 Docker Hub 官方镜像源发生网络波动。
- **解决办法**：修改 `docker-compose.yml` 中的 `image: postgres:15-alpine` 为国内镜像源 `docker.m.daocloud.io/library/postgres:15-alpine` 即可。
