# 接口设计说明书

## 1. 引言

### 1.1 编写目的

本文档描述"Golang AI 原生全栈应用快速开发脚手架"（以下简称"Go Scaffold"）的内部模块接口、外部服务接口以及生成项目的 API 接口设计。接口设计是系统实现的基础，确保各模块之间能够协同工作，并为后续编码和测试提供明确规范。

### 1.2 适用范围

本文档适用于以下范围：

1. 脚手架 CLI 工具的命令行接口；
2. 脚手架内部各功能模块之间的调用接口；
3. 脚手架与外部服务（如 Ollama）的通信接口；
4. 脚手架生成的后端项目对外暴露的 RESTful API 接口；
5. 脚手架生成项目与数据库的交互接口。

### 1.3 术语定义

| 术语 | 说明 |
|:---|:---|
| 脚手架 / Go Scaffold | 指命令行工具本身，运行在开发者本地，无状态、无用户体系 |
| 生成项目 | 指由脚手架渲染模板后输出的后端/前端项目，由开发者后续独立运行和维护 |
| 模板 | 指位于脚手架内部 `templates/` 目录下的文本文件，用于渲染生成项目代码 |
| RESTful API | 基于 HTTP 协议、符合 REST 架构风格的 API |
| JSON | JavaScript Object Notation，轻量级数据交换格式 |
| HTTP 方法 | GET、POST、PUT、DELETE、PATCH 等 |
| Status Code | HTTP 响应状态码 |
| Handler | HTTP 请求处理函数 |
| Middleware | 中间件，用于在请求处理前后执行通用逻辑 |

## 2. CLI 命令行接口

### 2.1 命令总览

脚手架 CLI 的入口为 `go-scaffold`，支持以下命令：

| 命令 | 功能 | 示例 |
|:---|:---|:---|
| `go-scaffold init` | 初始化新项目 | `go-scaffold init my-api` |
| `go-scaffold generate` | 基于自然语言生成代码 | `go-scaffold generate "user CRUD"` |
| `go-scaffold build` | 构建生成的项目 | `go-scaffold build --docker` |
| `go-scaffold --help` | 显示全局帮助 | `go-scaffold --help` |

### 2.2 `init` 命令接口

#### 2.2.1 命令格式

```bash
go-scaffold init <project-name> [flags]
```

#### 2.2.2 参数说明

| 参数 | 类型 | 必填 | 默认值 | 说明 |
|:---|:---|:---:|:---|:---|
| `project-name` | string | 是 | - | 项目名称，将作为目录名和模块名的一部分 |
| `--backend` | string | 否 | echo | 后端框架，可选值：echo、fiber、gin |
| `--orm` | string | 否 | ent | ORM 框架，可选值：ent、sqlc、gorm |
| `--db` | string | 否 | postgres | 数据库，可选值：postgres、mysql、sqlite |
| `--frontend` | string | 否 | react | 前端框架，可选值：react、vue、svelte |
| `--jwt` | bool | 否 | false | 是否启用 JWT 认证 |
| `--docker` | bool | 否 | false | 是否生成 Docker 配置 |
| `--ci` | bool | 否 | false | 是否生成 GitHub Actions 配置 |
| `--module-prefix` | string | 否 | github.com/example | Go 模块路径前缀 |
| `--interactive` | bool | 否 | true | 是否启用交互式 TUI |

#### 2.2.3 输出说明

命令执行成功后，输出如下：

```text
✓ 项目 my-api 创建成功
  后端框架: Echo
  ORM: Ent
  数据库: PostgreSQL
  前端框架: React 19 + TypeScript + Vite
  模块路径: github.com/example/my-api

后续步骤:
  cd my-api
  go mod tidy
  docker-compose up -d db
  go run ./cmd/server
```

### 2.3 `generate` 命令接口

#### 2.3.1 命令格式

```bash
# 启动 TUI 代码生成界面
go-scaffold generate

# 提供自然语言描述后进入 TUI，描述已预填充
go-scaffold generate "<description>"
```

#### 2.3.2 参数说明

| 参数 | 类型 | 必填 | 默认值 | 说明 |
|:---|:---|:---:|:---|:---|
| `description` | string | 否 | - | 自然语言描述，说明需要生成的代码；提供时 TUI 已预填充，用户可继续编辑 |
| `--model` | string | 否 | ornith:9b | 指定使用的 Ollama 模型 |
| `--dry-run` | bool | 否 | false | 仅显示生成结果，不写入文件 |

> 本命令采用 TUI 交互式界面收集生成所需信息（代码类型、实体名称、字段列表、输出路径等），不通过命令行 flag 暴露这些选项。

#### 2.3.3 TUI 代码生成界面

执行 `generate` 后进入专门的代码生成表单界面，包含以下元素：

```text
┌─────────────────────────────────────────┐
│  AI 代码生成                             │
│                                          │
│  描述: [生成用户 CRUD，包含 JWT 鉴权]      │
│  类型: [handler ▼]                       │
│  实体: [User                    ]        │
│  字段: [id:int,name:string,email:string] │
│                                          │
│  [预览]  [生成]  [取消]                   │
└─────────────────────────────────────────┘
```

交互行为：

1. **描述**：多行文本输入框，支持用户输入自然语言需求；若命令行已提供则预填充；
2. **类型**：下拉选择，可选值 `auto`、`handler`、`model`、`service`、`route`、`test`，默认 `auto`；
3. **实体**：单行文本输入，用于指定模型/实体名称，如 `User`、`Product`；
4. **字段**：单行文本输入，格式为 `name:type` 逗号分隔，如 `id:int,name:string`；TUI 实时解析并在下方显示字段预览；
5. **按钮**：
   - `预览`：调用 AI 生成结果并在 TUI 中展示，不写入文件（等效 `--dry-run`）；
   - `生成`：确认生成并写入项目；
   - `取消`：退出 TUI，不做任何修改。

字段类型支持：

| 简写类型 | 映射的 Go 类型 | 说明 |
|:---|:---|:---|
| `int` | `int` | 整型 |
| `int64` | `int64` | 长整型 |
| `string` | `string` | 字符串 |
| `bool` | `bool` | 布尔 |
| `time` | `time.Time` | 时间 |
| `float` | `float64` | 浮点 |

#### 2.3.4 输出说明

命令执行成功后，输出如下：

```text
✓ 代码生成成功
  实体: User
  类型: handler
  输出: internal/handler/user_handler.go

已生成文件:
  - internal/handler/user_handler.go
  - ent/schema/user.go
```

### 2.4 `build` 命令接口

#### 2.4.1 命令格式

```bash
go-scaffold build [flags]
```

#### 2.4.2 参数说明

| 参数 | 类型 | 必填 | 默认值 | 说明 |
|:---|:---|:---:|:---|:---|
| `--docker` | bool | 否 | false | 是否在生成项目内执行 `make docker-build` |
| `--output` | string | 否 | ./bin | 二进制文件输出目录 |
| `--target` | string | 否 | server | 构建目标，可选值：server、web、all |

#### 2.4.3 说明

`go-scaffold build` 是脚手架 CLI 对生成项目的构建代理：

- 默认调用生成项目目录下的 `make build`；
- `--docker` 时调用生成项目目录下的 `make docker-build`，镜像构建逻辑由生成项目的 Makefile 和 Dockerfile 负责；
- 脚手架自身不直接编译容器镜像，也不管理生成项目的运行时容器。

### 3.2 配置管理接口

#### 3.2.1 Config 结构体

```go
// Config 定义脚手架 CLI 自身的全局配置
// 注意：该配置仅作用于脚手架工具运行时，与生成项目的 .env 配置无关
type Config struct {
    DefaultBackend   string `mapstructure:"default_backend"`   // 默认后端框架
    DefaultORM       string `mapstructure:"default_orm"`       // 默认 ORM
    DefaultDatabase  string `mapstructure:"default_database"`  // 默认数据库
    DefaultFrontend  string `mapstructure:"default_frontend"`  // 默认前端框架
    OllamaHost       string `mapstructure:"ollama_host"`       // 本地 Ollama 地址
    OllamaModel      string `mapstructure:"ollama_model"`      // 默认模型，如 ornith:9b
    LogLevel         string `mapstructure:"log_level"`         // 脚手架日志级别
    ModulePrefix     string `mapstructure:"module_prefix"`     // 生成项目模块前缀
}
```

#### 3.2.2 接口方法

```go
// LoadConfig 从配置文件、环境变量和命令行参数加载配置
func LoadConfig(paths ...string) (*Config, error)

// SaveConfig 保存配置到文件
func SaveConfig(cfg *Config, path string) error

// MergeConfig 合并多个配置源，优先级：命令行 > 环境变量 > 配置文件
func MergeConfig(base *Config, overrides ...*Config) *Config
```

#### 3.2.3 配置说明

- `Config` 仅作用于脚手架 CLI 自身运行时，默认配置文件路径为 `~/.go-scaffold/config.yaml`；
- `ModulePrefix` 对应命令行参数 `--module-prefix`，默认值为 `github.com/example`；
- 若配置文件中未设置 `module_prefix`，`LoadConfig` 应保证返回的 `Config.ModulePrefix` 与默认值一致；
- 生成的项目模块路径格式为 `<ModulePrefix>/<ProjectName>`；
- 生成项目运行时的数据库、JWT 密钥等配置由生成项目自己的 `.env` 管理，与 `Config` 无关。

### 3.5 AI 代码生成接口

> **概述**：脚手架调用 Ollama `/api/chat` 端点，携带 `write_file` 工具定义。模型以 Tool Calling 方式返回结构化的文件写入指令，脚手架解析后执行写入并通过 `go/ast` 校验语法。整个过程为单次请求-响应，无多轮对话循环。

#### 3.5.1 GenerateInput 与项目上下文

```go
// GenerateInput 定义 AI 代码生成输入
// 注意：技术栈（Backend / ORM / Database）不应由用户每次生成时重复指定，
// 而应从当前项目的 .go-scaffold.json 元数据中读取并作为上下文注入 Prompt。
type GenerateInput struct {
    Description string   // 用户自然语言描述
    CodeType    string   // 生成代码类型
    Entity      string   // 实体名称
    Fields      []Field  // 实体字段列表
}

type Field struct {
    Name string
    Type string
    Tag  string
}
```

> 模型输出采用 Tool Calling 协议：通过 Ollama `/api/chat` + `tools` 参数控制输出。模型返回 `tool_calls` 数组，每个元素为一个 `write_file` 调用。本方案不使用多轮对话循环，仅为将输出格式从 JSON 文本替换为工具调用——本质上仍是单次生成。

#### 3.5.2 WriteFileTool 与输出结构体

```go
// WriteFileArgs —— write_file 工具的参数结构
// 对应 Ollama tools 定义中 write_file function 的 parameters schema
type WriteFileArgs struct {
    Path    string `json:"path"`    // 文件相对路径，如 internal/handler/user_handler.go
    Content string `json:"content"` // 完整的 Go 源代码文件内容
}

// CodeFile 定义单个生成文件
type CodeFile struct {
    Path    string // 文件相对路径
    Content string // 完整的文件内容
}

// GeneratedCode 定义 AI 代码生成输出
type GeneratedCode struct {
    Files []CodeFile // 生成的代码文件列表
}
```

**write_file 工具的 JSON Schema 定义**（作为 Ollama `tools` 参数传入）：

```json
{
  "type": "function",
  "function": {
    "name": "write_file",
    "description": "创建一个新的 Go 源代码文件。path 为相对于项目根目录的路径，content 为完整可编译的源代码，须包含 package 声明和所有 import。",
    "parameters": {
      "type": "object",
      "properties": {
        "path": {
          "type": "string",
          "description": "文件相对路径，如 ent/schema/user.go"
        },
        "content": {
          "type": "string",
          "description": "完整的 Go 源代码，包含 package 声明、import、错误处理和 slog 日志"
        }
      },
      "required": ["path", "content"]
    }
  }
}
```

#### 3.5.3 接口方法

```go
// NewCodeGenerator 创建代码生成器
func NewCodeGenerator(client *ollama.Client, model string) *CodeGenerator

// Generate 根据输入和项目元数据生成代码
//
// 采用 Tool Calling 单次生成协议：
//   1. 从 meta 读取项目技术栈（Backend / ORM / Database / ModulePath）作为上下文；
//   2. 构建 System Prompt + User Prompt，通过 /api/chat 发送，携带 write_file 工具定义；
//   3. 模型返回 tool_calls，每个 tool_call 为一次 write_file 调用；
//   4. 将 tool_calls 参数反序列化为 WriteFileArgs，组装为 GeneratedCode；
//   5. 对每个 file.content 执行 go/parser 语法校验。
func (g *CodeGenerator) Generate(ctx context.Context, input *GenerateInput, meta *ProjectMetadata) (*GeneratedCode, error)

// buildChatMessages 构建 /api/chat 的 messages 数组
func (g *CodeGenerator) buildChatMessages(input *GenerateInput, meta *ProjectMetadata) ([]ChatMessage, error)

// parseToolCalls 将模型返回的 tool_calls 解析为 GeneratedCode
func (g *CodeGenerator) parseToolCalls(toolCalls []ToolCall) (*GeneratedCode, error)

// ValidateSyntax 校验生成代码的语法正确性
func (g *CodeGenerator) ValidateSyntax(code string) error
```

#### 3.5.4 重试与错误处理

```go
// RetryPolicy 定义 AI 生成失败时的重试策略
type RetryPolicy struct {
    MaxRetries    int           // 最大重试次数，默认 3
    Backoff       time.Duration // 初始退避间隔
    BackoffFactor float64       // 退避倍数，默认 2.0
}

// GenerateWithRetry 带重试的代码生成
func (g *CodeGenerator) GenerateWithRetry(
    ctx context.Context,
    input *GenerateInput,
    meta *ProjectMetadata,
    policy *RetryPolicy,
) (*GeneratedCode, error)
```

重试触发条件：

1. Ollama 服务未启动或连接失败；
2. 模型返回的 `tool_calls` 列表为空或 `write_file` 调用数量为 0；
3. `tool_calls` 中的 `path` 或 `content` 参数缺失、类型错误（无法反序列化）；
4. 生成代码语法校验失败（`go/parser` 报错）。

超过最大重试次数后返回最终错误，并附带重试历史与排查指引。错误类型统一映射到 3.9.2 的 `InvalidOutputError`（1010）或 `OllamaConnectionError`（1008）。

#### 4.2.2 Prompt 构建规则

System Prompt 应包含以下要素：

1. 角色设定：明确模型作为 Go 后端开发专家；
2. 上下文信息：从 `ProjectMetadata` 读取当前项目使用的后端框架、ORM、数据库、模块路径，注入 System Prompt；
   - 注意：技术栈是项目级属性，由 `init` 阶段或 TUI 确定并写入 `.go-scaffold.json`，不应在每次 `generate` 时由用户重复指定；
3. 工具使用指令：明确要求模型使用 `write_file` 工具为每个新文件调用一次；
4. 输出规范：`content` 为完整可编译的 Go 源代码，包含包声明、所有 import、错误处理和 slog 日志；不要包含 Markdown 代码围栏。

User Prompt 包含：
1. 用户输入的自然语言描述；
2. 项目当前已有 schema 或 handler 摘要（如有，便于模型了解命名风格）。

#### 4.2.3 框架与组件约束

Prompt 应根据 `ProjectMetadata` 中的技术栈，向模型注入对应的框架使用约束。这些约束属于**生成项目**的代码规范，不是脚手架 CLI 自身的实现细节。

**Web 框架约束**

| 框架 | 约束要求 |
|:---|:---|
| Echo | handler 可返回 `error`，由 Echo 集中处理；路由注册使用 `e.GET/POST/PUT/DELETE`；中间件使用 `echo.MiddlewareFunc`。 |
| Fiber | handler 使用 `fiber.Ctx`；返回 `error` 或 `c.JSON`；路由注册使用 `app.Get/Post/Put/Delete`。 |
| Gin | handler 使用 `*gin.Context`；路由注册使用 `r.GET/POST/PUT/DELETE`；JSON 响应使用 `c.JSON`。 |

> 若项目同时选用 Huma，则优先要求 AI 使用 Huma 的 `huma.Register` 注册路由，输入输出定义为 struct，由 Huma 绑定到具体 Web 框架，AI 不直接写框架原生路由注册代码。

**ORM 约束**

| ORM | 约束要求 |
|:---|:---|
| Ent | AI 生成 `ent/schema/<entity>.go`，使用 `ent.Schema`、`field.*`、`index.*`；禁止手写 SQL 字符串；业务代码通过 `client.<Entity>.Create/Query/Update/Delete` 操作。 |
| sqlc | AI 生成 `queries/<entity>.sql` 和对应调用代码；业务代码调用 sqlc 生成的类型安全接口，禁止手写 SQL 拼接。 |
| GORM | AI 生成 `model/<entity>.go` struct 并嵌入 `gorm.Model`（可选）；业务代码使用 `db.Create/First/Find/Updates/Delete`。 |

**API 规范与文档约束（Huma）**

- AI 生成 handler 时，输入输出必须使用 struct，字段使用 `json` tag；
- 使用 `huma.Register` 注册路由，通过 `summary`、`operationId` 等标签生成 OpenAPI 元数据；
- 禁止 AI 手写 swagger/yaml 注释，OpenAPI 文档由 Huma 从 struct 推导；
- 错误返回统一使用 `huma.Error400/BadRequest` 等标准错误构造器。

**可观测性约束（log/slog）**

- 所有错误处理必须使用 `slog.ErrorContext(ctx, "...", slog.Any("err", err))` 记录；
- 关键业务操作使用 `slog.InfoContext(ctx, "...", slog.Any("key", value))`；
- 禁止使用 `fmt.Println`、`log.Printf` 或 `log.Println`；
- 日志字段使用结构化键值对，便于后续接入 OpenTelemetry。

**数据库约束**

- 根据 `Database` 字段在 Prompt 中声明目标数据库类型（PostgreSQL / MySQL / SQLite）；
- 生成的 Ent schema / sqlc SQL / GORM tag 应符合对应数据库的语法和类型约定；
- 时间字段统一使用 `time.Time`，由 ORM 处理存储格式。
