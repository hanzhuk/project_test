# 接口设计说明书

## 1. 引言

### 1.1 编写目的

本文档描述"Golang AI 原生全栈应用快速开发脚手架"（以下简称"Go Scaffold"）的内部模块接口、外部服务接口以及生成项目的 API 接口设计。接口设计是系统实现的基础，确保各模块之间能够协同工作，并为后续编码和测试提供明确规范。

### 1.2 适用范围

本文档适用于以下范围：

1. 脚手架 CLI 工具的命令行接口；
2. 脚手架内部各功能模块之间的调用接口；
3. 脚手架与外部服务（如 Ollama）的通信接口；
4. 脚手架生成的后端项目对外暴露的 RESTful API 接口；
5. 脚手架生成项目与数据库的交互接口。

### 1.3 术语定义

| 术语 | 说明 |
|:---|:---|
| RESTful API | 基于 HTTP 协议、符合 REST 架构风格的 API |
| JSON | JavaScript Object Notation，轻量级数据交换格式 |
| HTTP 方法 | GET、POST、PUT、DELETE、PATCH 等 |
| Status Code | HTTP 响应状态码 |
| Handler | HTTP 请求处理函数 |
| Middleware | 中间件，用于在请求处理前后执行通用逻辑 |

## 2. CLI 命令行接口

### 2.1 命令总览

脚手架 CLI 的入口为 `go-scaffold`，支持以下命令：

| 命令 | 功能 | 示例 |
|:---|:---|:---|
| `go-scaffold init` | 初始化新项目 | `go-scaffold init my-api` |
| `go-scaffold generate` | 基于自然语言生成代码 | `go-scaffold generate "user CRUD"` |
| `go-scaffold build` | 构建生成的项目 | `go-scaffold build --docker` |
| `go-scaffold --help` | 显示全局帮助 | `go-scaffold --help` |

### 2.2 `init` 命令接口

#### 2.2.1 命令格式

```bash
go-scaffold init <project-name> [flags]
```

#### 2.2.2 参数说明

| 参数 | 类型 | 必填 | 默认值 | 说明 |
|:---|:---|:---:|:---|:---|
| `project-name` | string | 是 | - | 项目名称，将作为目录名和模块名的一部分 |
| `--backend` | string | 否 | echo | 后端框架，可选值：echo、fiber、gin |
| `--orm` | string | 否 | ent | ORM 框架，可选值：ent、sqlc、gorm |
| `--db` | string | 否 | postgres | 数据库，可选值：postgres、mysql、sqlite |
| `--frontend` | string | 否 | react | 前端框架，可选值：react、vue、svelte |
| `--jwt` | bool | 否 | false | 是否启用 JWT 认证 |
| `--docker` | bool | 否 | false | 是否生成 Docker 配置 |
| `--ci` | bool | 否 | false | 是否生成 GitHub Actions 配置 |
| `--module-prefix` | string | 否 | github.com/example | Go 模块路径前缀 |
| `--interactive` | bool | 否 | true | 是否启用交互式 TUI |

#### 2.2.3 输出说明

命令执行成功后，输出如下：

```text
✓ 项目 my-api 创建成功
  后端框架: Echo
  ORM: Ent
  数据库: PostgreSQL
  前端框架: React 19 + TypeScript + Vite
  模块路径: github.com/example/my-api

后续步骤:
  cd my-api
  go mod tidy
  docker-compose up -d db
  go run ./cmd/server
```

### 2.3 `generate` 命令接口

#### 2.3.1 命令格式

```bash
go-scaffold generate "<description>" [flags]
```

#### 2.3.2 参数说明

| 参数 | 类型 | 必填 | 默认值 | 说明 |
|:---|:---|:---:|:---|:---|
| `description` | string | 是 | - | 自然语言描述，说明需要生成的代码 |
| `--type` | string | 否 | auto | 生成代码类型，可选值：auto、handler、model、service、route、test |
| `--output` | string | 否 | - | 输出文件路径，默认根据类型自动推断 |
| `--entity` | string | 否 | - | 实体名称，如 User、Product |
| `--fields` | string | 否 | - | 实体字段列表，逗号分隔，如 `id:int,name:string` |
| `--model` | string | 否 | qwen2.5-coder:7b | 指定使用的 Ollama 模型 |
| `--dry-run` | bool | 否 | false | 仅显示生成结果，不写入文件 |

#### 2.3.3 输出说明

命令执行成功后，输出如下：

```text
✓ 代码生成成功
  实体: User
  类型: handler
  输出: internal/handler/user_handler.go

已生成文件:
  - internal/handler/user_handler.go
  - ent/schema/user.go
```

## 3. 内部模块接口

### 3.1 模块划分

脚手架内部按职责划分为以下模块：

| 模块 | 包路径 | 职责 |
|:---|:---|:---|
| cmd | `cmd/go-scaffold` | CLI 入口和命令注册 |
| command | `internal/command` | 命令执行逻辑 |
| config | `internal/config` | 配置读取与管理 |
| template | `internal/template` | 模板加载与渲染 |
| project | `internal/project` | 项目结构生成 |
| ai | `internal/ai` | AI 代码生成 Flow |
| build | `internal/build` | 项目构建与打包 |
| log | `internal/log` | 日志管理 |

### 3.2 配置管理接口

#### 3.2.1 Config 结构体

```go
// Config 定义脚手架的全局配置
type Config struct {
    DefaultBackend   string `mapstructure:"default_backend"`
    DefaultORM       string `mapstructure:"default_orm"`
    DefaultDatabase  string `mapstructure:"default_database"`
    DefaultFrontend  string `mapstructure:"default_frontend"`
    OllamaHost       string `mapstructure:"ollama_host"`
    OllamaModel      string `mapstructure:"ollama_model"`
    LogLevel         string `mapstructure:"log_level"`
    ModulePrefix     string `mapstructure:"module_prefix"`
}
```

#### 3.2.2 接口方法

```go
// LoadConfig 从配置文件、环境变量和命令行参数加载配置
func LoadConfig(paths ...string) (*Config, error)

// SaveConfig 保存配置到文件
func SaveConfig(cfg *Config, path string) error

// MergeConfig 合并多个配置源，优先级：命令行 > 环境变量 > 配置文件
func MergeConfig(base *Config, overrides ...*Config) *Config
```

## 4. 外部服务接口

### 4.1 Ollama API 接口

#### 4.1.1 接口地址

- 默认地址：`http://localhost:11434`
- 生成接口：`POST /api/generate`

#### 4.1.2 请求参数

| 字段 | 类型 | 必填 | 说明 |
|:---|:---|:---:|:---|
| model | string | 是 | 模型名称，如 `qwen2.5-coder:7b` |
| prompt | string | 是 | 发送给模型的提示文本 |
| stream | bool | 否 | 是否流式返回，本系统使用 `false` |
| format | string | 否 | 输出格式，可指定 `json` |
| options | object | 否 | 模型推理参数 |

#### 4.1.3 Options 参数

| 字段 | 类型 | 默认值 | 说明 |
|:---|:---|---:|:---|
| temperature | float | 0.2 | 采样温度，越低输出越确定 |
| top_p | float | 0.9 | 核采样参数 |
| top_k | int | 40 | Top-K 采样参数 |
| num_predict | int | 2048 | 最大生成 token 数 |

#### 4.1.4 响应参数

| 字段 | 类型 | 说明 |
|:---|:---|:---|
| model | string | 使用的模型名称 |
| created_at | string | 响应创建时间 |
| response | string | 模型生成的文本内容 |
| done | bool | 是否生成完成 |
| total_duration | int64 | 总耗时，单位为纳秒 |
| load_duration | int64 | 模型加载耗时 |
| prompt_eval_count | int | Prompt token 数 |
| eval_count | int | 生成 token 数 |

### 4.2 Genkit Go Flow 接口

#### 4.2.1 Flow 定义

```go
// 定义代码生成 Flow
codeGenFlow := genkit.DefineFlow(g, "codeGenerator",
    func(ctx context.Context, input *GenerateInput) (*GeneratedCode, error) {
        prompt := buildPrompt(input)
        resp, err := ollamaClient.Generate(ctx, model, prompt)
        if err != nil {
            return nil, err
        }
        return parseGeneratedCode(resp.Response)
    })
```

#### 4.2.2 Prompt 构建规则

Prompt 模板应包含以下要素：

1. 角色设定：明确模型作为 Go 后端开发专家；
2. 上下文信息：项目使用的后端框架、ORM、数据库；
3. 任务描述：用户输入的自然语言描述；
4. 输出格式要求：仅返回代码、使用指定包名、包含错误处理；
5. Few-shot 示例：提供 1-2 个标准代码示例。

## 5. 生成项目的 RESTful API 接口

### 5.1 接口设计原则

生成的后端项目默认提供 RESTful API，遵循以下原则：

1. 使用资源路径表示实体，如 `/users`、`/products`；
2. 使用 HTTP 方法表示操作：GET（查询）、POST（创建）、PUT（更新）、DELETE（删除）；
3. 统一返回 JSON 格式响应；
4. 使用标准 HTTP 状态码；
5. 错误响应包含 `code`、`message`、`details` 字段。

### 5.2 统一响应格式

#### 5.2.1 成功响应

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

#### 5.2.2 错误响应

```json
{
  "code": 1001,
  "message": "请求参数错误",
  "details": ["用户名不能为空"]
}
```

### 5.3 用户管理接口示例

#### 5.3.1 创建用户

- **接口**：`POST /api/v1/users`
- **Content-Type**：`application/json`
- **请求体**：

```json
{
  "username": "alice",
  "email": "alice@example.com",
  "password": "SecurePass123"
}
```

```json
{

    "username": "alice",

}
```

#### 5.3.2 查询用户列表

- **查询参数**：

| 参数 | 类型 | 必填 | 说明 |
|:---|:---|:---:|:---|
| page | int | 否 | 页码，默认 1 |
| page_size | int | 否 | 每页数量，默认 10 |
| keyword | string | 否 | 搜索关键词 |

- **响应体（200 OK）**：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "list": [
      {
        "id": 1,

      }

    "page": 1,

```

#### 5.3.3 查询单个用户

### 5.5 状态码定义

| HTTP 状态码 | 含义 | 使用场景 |
|:---:|:---|:---|
| 200 | OK | 查询、更新成功 |
| 201 | Created | 创建成功 |
| 204 | No Content | 删除成功 |
| 400 | Bad Request | 请求参数错误 |
| 401 | Unauthorized | 未提供有效认证信息 |
| 403 | Forbidden | 无权限访问 |
| 404 | Not Found | 资源不存在 |
| 409 | Conflict | 资源冲突，如重复邮箱 |
| 422 | Unprocessable Entity | 语义错误 |
| 500 | Internal Server Error | 服务器内部错误 |

### 5.6 业务错误码定义

| 错误码 | 含义 | HTTP 状态码 |
|:---:|:---|:---:|
| 0 | 成功 | 200/201/204 |
| 1001 | 参数校验失败 | 400 |
| 1002 | 资源不存在 | 404 |
| 1003 | 资源已存在 | 409 |
| 1004 | 认证失败 | 401 |
| 1005 | 权限不足 | 403 |
| 1006 | 服务器内部错误 | 500 |
| 1007 | 数据库操作失败 | 500 |

## 6. 数据库接口

### 6.1 Ent 接口

生成的后端项目使用 Ent 作为 ORM，通过代码生成获得类型安全的数据库客户端。

#### 6.1.1 客户端创建

```go
import "entgo.io/ent/dialect/sql"

// 创建数据库驱动
drv, err := sql.Open("postgres", dsn)
if err != nil {
    return err
}

// 创建 Ent 客户端
client := ent.NewClient(ent.Driver(drv))
defer client.Close()
```

#### 6.1.2 常用 CRUD 接口

```go
// 创建记录
user, err := client.User.Create().
    SetUsername("alice").
    SetEmail("alice@example.com").
    SetPasswordHash(hash).
    Save(ctx)

// 查询单条
user, err := client.User.Get(ctx, id)

// 查询列表
users, err := client.User.Query().
    Where(user.UsernameContains("ali")).
    Offset(offset).
    Limit(limit).
    All(ctx)

// 更新
user, err := client.User.UpdateOneID(id).
    SetUsername("alice_updated").
    Save(ctx)

// 删除
err := client.User.DeleteOneID(id).Exec(ctx)
```

### 6.2 数据库连接配置

生成的项目通过环境变量配置数据库连接：

| 环境变量 | 说明 | 示例 |
|:---|:---|:---|
| DB_HOST | 数据库主机地址 | localhost |
| DB_PORT | 数据库端口 | 5432 |
| DB_USER | 数据库用户名 | postgres |
| DB_PASSWORD | 数据库密码 | secret |
| DB_NAME | 数据库名称 | myapi_db |
| DB_SSL_MODE | SSL 模式 | disable |

## 7. 前端与后端交互接口

### 7.1 API 客户端生成

生成的 React 前端项目使用 OpenAPI 3.1 规范文件，通过 `openapi-typescript` 生成 TypeScript 类型定义，并通过 `openapi-fetch` 或自定义 fetch 封装调用后端 API。

### 7.2 示例调用

```typescript
// 使用 openapi-fetch 调用后端 API
import createClient from 'openapi-fetch';
import type { paths } from './api-types';

const client = createClient<paths>({ baseUrl: import.meta.env.VITE_API_BASE_URL });

async function createUser(user: { username: string; email: string; password: string }) {
  const { data, error } = await client.POST('/api/v1/users', {
    body: user,
  });
  if (error) throw error;
  return data;
}
```
